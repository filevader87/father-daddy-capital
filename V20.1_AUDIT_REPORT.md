## 3. SIGNAL COMPUTATION: enhanced_signal()

### 3.1 RSI Computation
- **Period**: 14 candles (standard Wilder RSI)
- **Source**: 5-minute candles from Polymarket CLOB
- **Direction Mapping**:
  - `RSI < 35` → `direction = "UP"` (oversold, expect bounce up)
  - `RSI > 65` → `direction = "DOWN"` (overbought, expect reversion down)
  - `35 ≤ RSI ≤ 65` → **SKIP** (neutral, no signal)

### 3.2 Confidence Scoring
- Base confidence from RSI extremity: `confidence = abs(RSI - 50) / 50`
- RSI at 28.3 → confidence = 0.85 (observed in sim)
- RSI at 27.7 → confidence = 0.85 (observed in sim)

### 3.3 Token State Classification
- Uses `classify_token_state()` to tag BTC as uptrend/downtrend/neutral
- Checks candle data: `open[-1] < open[-5]` → uptrend
- Calls `compute_downtrend_veto()` for additional downtrend validation
- **Downtrend veto threshold**: `open > 1.5 × low_20` (magic number, not configurable)

### 4.2 Orderbook Transition Gate

- Final clamped to **[-1.0, +1.0]**

## 6. SETTLEMENT & PnL

### 6.1 Settlement Flow
```
Every cycle:
  For each open position:
    Check if contract has expired (5min window)
    If expired:
      Fetch result from Gamma API
      Determine win/loss (UP token resolved to 1.0 or 0.0)
      Calculate PnL: 
        UP win:  payout = (1.0 - entry_price) × size → profit
        UP loss:  payout = 0 → loss = entry_price × size
        DOWN win: payout = entry_price × size → profit
        DOWN loss: payout = 0 → loss = (1.0 - entry_price) × size
      Close position in tracker
      Log CLOSE event
```

### 6.2 PnL Math Verification
- Entry at 0.50, $2 size → cost = $1.00 per side
- UP win at 0.50: payout = (1.0 - 0.50) × 2 = $1.00 profit
- UP loss at 0.50: loss = 0.50 × 2 = $1.00 loss
- DOWN win at 0.50: payout = 0.50 × 2 = $1.00 profit
- DOWN loss at 0.50: loss = (1.0 - 0.50) × 2 = $1.00 loss
- **At 0.50, win and loss are symmetric: ±$1.00**

**File**: `src/sentiment/xai_x_sentiment.py`, line 338
```python

**Impact**: If sentiment is re-enabled, the veto gate will NEVER block UP bounces — even during panic events. The raw sentiment dict will be returned instead of the `{"veto": True/False, "veto_reason": ...}` structure.
**Fix**: Change to `.get_veto_context(asset_key)`
**Risk**: Currently mitigated because sentiment is DISABLED. But this is a latent bomb.

### 🟡 MEDIUM: Gate trivially true in paper mode
**File**: `pm_engine_v19_8.py`, line ~3353

gates["book_executable_working"] = len(state.get("positions", {})) >= 0  # Always True
```
**Impact**: This readiness gate always passes — it doesn't actually validate that the book is executable.
**Fix**: Should check `>= 1` after first trade, or validate against real book depth.

### 🟡 MEDIUM: Probability lag state classification overlap
**File**: `src/microstructure/probability_lag.py`, lines ~153-164

- The code checks `repricing_lag` first, so `underreaction` is unreachable in that overlap zone
- **Impact**: Underreaction trades that should be confirmed may be misclassified as repricing_lag. Currently irrelevant since lag gate is DISABLED.
